# PWA Conversion Roadmap

This roadmap outlines the architecture and implementation steps required to upgrade the Indian Financial News Aggregator's Next.js frontend into a production-grade Progressive Web Application (PWA).

## 1. PWA Integration Strategy

We will utilize `@ducanh2912/next-pwa` (a maintained fork of `next-pwa` specifically designed for Next.js App Router architectures and Turbopack compatibility). This library wraps Workbox to automatically generate and register service workers during the Next.js build phase.

**Configuration Update (`next.config.ts`)**:
```typescript
const withPWA = require('@ducanh2912/next-pwa').default({
  dest: 'public',
  disable: process.env.NODE_ENV === 'development',
  register: true,
  skipWaiting: true,
});

module.exports = withPWA(nextConfig);
```

## 2. Web Manifest (`manifest.ts`)

Instead of a static `manifest.json`, we will leverage Next.js App Router's metadata API by creating `app/manifest.ts`. This dynamically generates the manifest at build time.

* **Name**: Indian Financial News Aggregator
* **Short Name**: FinNews
* **Display**: `standalone` (removes browser UI for a native app feel)
* **Background Color**: `#ffffff` (matches tailwind background)
* **Theme Color**: `#000000` (for status bar styling)

## 3. App Icons

We will generate a complete suite of standard PWA icons and place them in the `public/icons` directory:
* `icon-192x192.png` (Android/Chrome standard)
* `icon-512x512.png` (High-res splash)
* `apple-touch-icon.png` (Required for iOS Safari "Add to Home Screen" functionality)

## 4. Splash Screens

iOS requires explicitly defined startup images for splash screens, otherwise it shows a white flash. We will implement `apple-touch-startup-image` `<link>` tags in the global layout or utilize a library like `pwa-asset-generator` to auto-generate the necessary viewport-specific splash screens for all iOS devices.

## 5. Offline Caching Strategy

The service worker generated by Workbox will enforce a robust caching strategy:
* **Static Assets (CSS/JS/Fonts)**: `CacheFirst` strategy with long expiration.
* **Navigation Requests**: `NetworkFirst` falling back to an offline fallback page (`app/~offline/page.tsx`).
* **Images**: `StaleWhileRevalidate` with an upper bound limit to prevent Cache Storage bloat.

## 6. Article Caching (API Strategy)

Since financial news is dynamic, aggressive caching can result in stale data. We will configure the Service Worker runtime caching specifically for the `/articles` backend endpoint:
* **Strategy**: `StaleWhileRevalidate`
* **Mechanism**: When a user goes completely offline, they will see the articles from their last successful fetch.
* **Storage Limit**: Set a maximum of 50 API responses to preserve browser quota constraints.

## 7. Installability Validation

To meet Chrome and Safari installability heuristics, the application must:
1. Be served over HTTPS (handled in production by proxy/TLS).
2. Have a valid manifest with `name`, `icons`, and `start_url`.
3. Register a Service Worker with a functional `fetch` event handler.
4. Pass the Chrome DevTools "Application -> Manifest" criteria without warnings.

## 8. Lighthouse PWA Scoring

Our CI pipeline can be extended to include Lighthouse CLI checks (`@lhci/cli`). 
* Target Score: **100/100 PWA**
* We will automate a headless Chrome run against a spun-up Docker instance to verify the presence of the Service Worker, HTTPS redirects, and valid Apple Touch attributes.

## 9. Docker Deployment Impact

Converting to a PWA introduces exactly **zero** backend infrastructure changes and minimal Next.js Docker changes. 
* The `next build` command will output `sw.js` and `workbox-*.js` into the `.next/standalone/public` directory. 
* Since we are already using `output: 'standalone'` in our Next.js Dockerfile, these generated service worker assets will automatically be served statically without requiring additional NGINX configuration.
