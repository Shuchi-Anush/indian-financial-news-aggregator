# Release Notes — v1.4.0 Backend Stabilized

**Project:** Indian Financial News Aggregator
**Release Date:** 2026-06-21

# Overview

v1.4.0 focuses exclusively on backend stabilization, operational hardening, and technical debt reduction.

This release does not introduce new ingestion sources, frontend features, AI functionality, or user-facing capabilities.

Instead, the goal was to improve scalability, reliability, maintainability, security, and development workflow quality while preserving all existing API contracts.

---

# Objectives

The primary objectives of this release were:

* Eliminate confirmed scalability bottlenecks.
* Improve multi-instance deployment safety.
* Reduce architectural complexity.
* Improve test coverage.
* Secure administrative endpoints.
* Introduce automated validation pipelines.
* Prepare the backend for frontend development.

# Key Changes

## 1. Legacy Collector Cleanup

### Problem

The repository still contained legacy RSS collector wrapper classes that were no longer used after the migration to the dynamic `BaseRSSCollector` factory.

### Solution

Removed unused collector wrappers:

* `businessstandard.py`
* `cnbctv18.py`
* `economictimes.py`
* `livemint.py`
* `moneycontrol.py`
* `sebi.py`

### Result

* Reduced maintenance burden.
* Simplified collector architecture.
* Eliminated dead code.

---

## 2. Deduplication Scalability Refactor

### Previous Design

The ingestion pipeline loaded up to 100,000 historical URLs and content hashes into memory during deduplication.

This created:

* O(N) memory growth
* Increased startup cost
* Future scalability limitations

### New Design

The pipeline now performs targeted database lookups using only the URLs and hashes present in the incoming batch.

### Result

* Memory usage scales with batch size instead of database size.
* Improved long-term scalability.
* Preserved idempotent ingestion behavior.

---

## 3. Distributed Ingestion Protection

### Previous Design

The scheduler relied on an in-process `asyncio.Lock()`.

This only protected a single Python process.

### New Design

Integrated PostgreSQL advisory locking into ingestion orchestration.

### Result

* Safe for multi-process deployments.
* Safe for multi-container deployments.
* Prevents duplicate ingestion execution.
* Gracefully skips execution when another worker already owns the lock.

---

## 4. Administrative Endpoint Security

### Previous Design

Administrative endpoints were publicly accessible.

### New Design

Added API key authentication using:

* Environment-based configuration
* FastAPI dependency injection
* Request header validation

### Result

Protected endpoints include:

* Pipeline status
* Source health
* Operational monitoring endpoints
